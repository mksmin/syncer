# Remote index cache

## Короткий ответ

Хранить manifest с индексом файлов на Яндекс Диске технически возможно: официальный REST API
разрешает приложениям сохранять настройки и файлы в облаке. Но Syncer сейчас имеет read-only OAuth и
pull-only provider contract. Запись manifest потребует нового permission, повторной авторизации и
отдельного security design.

## Почему manifest не заменяет полный обход

- Изменения через сайт, desktop client или другое приложение не обновят manifest Syncer.
- Несколько устройств могут одновременно перезаписать manifest; транзакции между индексом и
  содержимым каталога нет.
- Crash во время upload оставит старый или повреждённый индекс.
- Проверка актуальности всех вложенных папок может снова потребовать полного обхода.
- Manifest раскрывает имена, размеры, даты и checksum файлов и занимает место на Диске.
- Любая remote write расширяет угрозы: баг pull-клиента впервые сможет изменить источник истины.

## Решение для текущей архитектуры

1. In-memory remote index с коротким TTL ускоряет повторный запуск в одной сессии.
2. Per-file snapshot в локальном `data.json` не скачивает повторно уже подтверждённые файлы.
3. Следующий безопасный шаг — локальный persistent index с TTL и обязательным периодическим полным
   scan. Он не меняет Яндекс Диск.
4. Remote manifest допустим только как необязательная подсказка, никогда как источник истины. При
   пропуске, конфликте или повреждении нужен полный scan.

Официальное описание REST API прямо допускает хранение настроек приложения на Диске:
<https://yandex.com/dev/disk/rest/?lang=ru-RU>. Это подтверждает техническую возможность, но не даёт
гарантированного change feed для произвольного дерева.
